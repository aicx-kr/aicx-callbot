## 1. 티켓 원문

### 배경/목적
현재 `logging.py` 는 단순 stdout 텍스트 포맷이며 `tracer.py` 의 TraceRecorder 도 운영 가시성이 부족하다. 이슈 발생 시 통화 단위로 디버깅이 불가능한 상태로, 구조화 로그 및 correlation ID 체계 구축이 필요하다.

### 작업 내용
- 구조화 로그(JSON) 포맷으로 전환 — `call_id`, `bot_id`, `tenant_id` 필드 포함
- 통화 세션 단위 correlation ID 전파 (`voice_session.py` → 모든 하위 서비스)
- STT / LLM / TTS 각 단계별 latency 로그 (`stt_ms`, `llm_ms`, `tts_ms`)
- 에러 레벨 로그 발생 시 Slack 알림 연결 (webhook 방식, 설정 없으면 비활성)

### 완료 기준
- 통화 1건의 전체 로그를 `call_id` 로 grep 가능
- 각 단계 latency 가 로그에 ms 단위로 기록됨
- 에러 로그 → Slack 알림 동작 (환경변수 `SLACK_WEBHOOK_URL` 설정 시)

## 4. 본 문서가 추가하는 결정

### 4.1 Slack 핸들러 정책
- 레벨: ERROR 이상만 (WARNING 은 발송 X)
- 위치: `backend/src/core/logging/handlers/slack.py` (기존 설계의 파일 트리에 추가)
- 활성화: `SLACK_WEBHOOK_URL` env 있을 때만. 없으면 핸들러 자체를 logger 에 attach 안 함
- **Rate limit 필수**: `error_signature = (logger, msg_template)` 키, 60초 윈도우당 1회. 같은 통화에서 같은 에러 100번 발생해도 1회만 Slack 전송 (LLM API 장애 같은 burst 시 채널 도배 방지)
- 메시지 포맷: `call_id`, `bot_id`, `error_signature`, 최근 5줄 컨텍스트 포함

### 4.2 tenant_id 의무화 + default
- callbot 의 멀티테넌트화 시점에 의무 필드로 승격
- 그 전에는 모든 로그 레코드에 `tenant_id="default"` 자동 주입 (ContextVar 기본값)
- 향후 멀티테넌트화 시 `set_system("tenant_id", ...)` 한 곳만 변경

### 4.3 민감정보 마스킹
- STT transcript 본문은 로그 메시지 본문에 절대 넣지 않음 (`stt.final` 이벤트의 `text` 필드만 별도 dedicated 로거에 — DB Trace 에는 저장 OK, 표준 로그 라인엔 X)
- `extracted` 필드 (개인정보 가능: 이름·전화번호·예약번호) 도 동일 정책

### 4.4 end_reason enum 의존
AICC-910 §5.3 에서 `CallSession.end_reason` 을 6값 Literal 로 표준화 — `call.end` 이벤트의 `reason` 필드와 동일 enum 사용. 본 티켓에서 enum 정의를 먼저 잡고 910 가 따라옴.

## 6. 마일스톤 출구 기준
- 통화 1건의 전체 로그를 `grep '"call_id":"<uuid>"'` 로 추적 가능
- `stt_ms` / `llm_ms` / `tts_ms` 가 JSON 필드로 분리 기록 (turn 당 1줄 또는 단계별 줄)
- `SLACK_WEBHOOK_URL` 설정 + 의도된 에러 발생 시 Slack 수신
- 같은 에러 100회 burst 시 Slack 1회만 (rate limit 동작)
- ContextVar 전파: WebSocket → asyncio.Task 분기까지 call_id 유실 0
