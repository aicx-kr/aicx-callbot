한 줄 요약

한 통화에 봇 여러 명이 협업한다. 그중 한 명은 메인(통화를 시작·자유 응대), 나머지는 서브(환불·결제 같은 절차 전문). 메인이 의도를 듣고 적절한 서브에게 넘기고, 서브가 끝나면 다시 메인으로 돌아온다.

4가지 핵심

1누구·무엇

한 통화 = 봇 묶음. 메인 1명 + 서브 N명. 각 봇은 자기만의 페르소나·지식·도구를 가짐.

2두 가지 방식

자유 대화를 잘하는 프롬프트 봇과, 절차를 빠짐없이 수행하는 플로우 봇. 한 봇은 둘 중 한 방식만.

3통화 흐름

매 발화마다 2단계 — ① 공통 규칙 검사 → ② 현재 활성 봇 실행. 핸드오프도 봇 실행 결과로 처리. 변수는 모두가 공유.

4코드 원칙

비즈니스 규칙은 한 곳에만. DB·외부 API 사용 부분과 명확히 분리.

1누구·무엇

한 통화 안에는 봇 한 명이 아니라 여러 명이 협업한다. 그리고 각 봇은 자기만의 자산(페르소나·스킬·지식·도구)을 가진다.

💡 한 통화 = 봇 묶음. 봇 한 명 = 페르소나 · 스킬 · 지식 자료 · 도구 4가지로 구성. 통화 단위 설정(음성·인사말 등)은 통화 자체에 붙어있어 한 통화 동안 일관됨.

2두 가지 일하는 방식

봇이 사용자 발화에 응답하는 방식은 두 가지. 한 봇은 둘 중 하나만 선택한다 (섞지 않음).

프롬프트 봇 — 자유 대화 잘함

어떻게 동작하나: 봇에게 "이런 일을 하는 친절한 안내원이야"라고 글로 적어두면, 사용자 발화에 LLM이 알아서 자연스럽게 답함.

잘 맞는 경우: 인사·FAQ·일반 응대·의도 파악. 어떤 질문이 올지 미리 모를 때.

약점: "이 3가지 정보는 무조건 받아라"같은 절차 강제가 어려움. LLM이 가끔 누락.

플로우 봇 — 절차를 빠짐없이

어떻게 동작하나: 봇의 응답을 노드로 그래프처럼 그려둠. "1단계: 이름 받기 → 2단계: 사유 받기 → 3단계: 확정". 사용자 발화가 와도 정해진 흐름대로만 진행.

잘 맞는 경우: 결제·본인확인·환불처럼 필드 누락이 사고로 이어지는 절차.

약점: 만드는 데 시간 들고, 사용자가 예상 밖 말을 하면 어색해질 수 있음.

핵심 패턴 — 메인은 프롬프트, 분기만 플로우

💡 첫 진입은 의도가 미정이라 프롬프트가 자연스럽다. 절차 필요한 분기만 플로우로 만들어 누락 방지. 분기가 끝나면 메인으로 돌아옴.

3분기가 필요할 때 vs 필요 없을 때

서브 봇(분기)이 없어도 콜봇은 잘 동작한다. 단순 안내·FAQ 봇은 메인 하나로 충분. 분기는 절차가 명시적으로 필요할 때만 추가.

경우 1 — 분기가 아예 없는 단순 콜봇

예: 단순 안내봇, FAQ 봇, 인사 봇. 한 통화 묶음 안에 메인 봇만 두면 됨. 모든 발화에 메인 봇이 응대하고 종료. → 빌더가 분기를 만들지 않으면 서브 봇은 0개, 시스템은 그대로 동작.

경우 2 — 분기는 있는데 이번 발화가 트리거 안 됨

예 발화: "위치가 어디예요?" → 공통 규칙 매치 없음 → 메인 봇이 자기 지식 자료(위치 안내)로 답변 생성 → "강남역 3번 출구에서 도보 5분입니다". 서브 봇은 건드리지도 않음. 트리거 매치되는 발화가 올 때까지 메인이 계속 응대.

경우 3 — 분기 트리거 매치 (메인 → 서브)

예 발화: "환불하고 싶어요" → 메인 봇 의도 분류 = "환불" → 환불 봇으로 핸드오프 (변수도 그대로 인계) → 환불 봇이 절차 진행 (주문번호 → 사유 → 환불 API → 확인 SMS) → 환불 봇 종료 → 메인 봇으로 복귀 → 다음 발화 대기.

💡 메인 봇이 기본 응답자. 트리거 안 맞으면 메인이 계속 함. 필요할 때만 서브가 잠깐 끼어들어 절차 처리하고 다시 메인으로. 분기 없이 메인 하나만 두는 것도 완전히 OK.

4통화 흐름 — 발화 한 번이 처리되는 2단계

사용자 발화 한 번이 시스템 안에서 거치는 2단계. 매 발화마다 ① 공통 규칙 검사가 항상 첫 단계이고, ② 봇 실행 결과에 따라 다음 활성 봇이 갱신된다.

핸드오프는 별도 단계가 아님

"메인 봇 → 환불 봇" 전환은 ② 봇 실행 결과의 일부로 처리된다. 메인 봇 LLM이 응답할 때 "환불 봇으로 넘기자"는 신호를 같이 내면, 다음 활성 봇이 환불 봇으로 갱신될 뿐. 별도 라우팅 서비스 없음.

내부 명칭 (개발자용): ① = Dispatcher · ② = Executor. Skill.is_frontdoor는 "이 스킬이 메인 진입점"이라는 플래그일 뿐, 별도 서비스 아님. 핸드오프는 봇 실행 결과의 후처리 (활성 봇 ID 갱신 + 변수 인계)로 단순화.

변수 컨테이너 — 통화 내내 공유되는 메모

2단계 내내 같은 변수 묶음이 공유된다. 메인 봇이 받은 이름을 환불 봇이 그대로 쓰고, 환불 봇이 채운 주문번호를 다음 봇이 참조 가능. 핸드오프해도 변수는 끊기지 않음. 변수 출처는 3가지:

  • ① 통화 시작 시 주입 — SDK·웹훅·아웃바운드 캠페인이 미리 넣어둠 (customer_name, phone 등)
  • ② 시스템 자동 — 통화 ID·시작 시각·발신 번호 등 시스템이 자동 채움
  • ③ 대화 중 추출 — 플로우 봇이 사용자 발화에서 뽑아냄 (주문번호·환불 사유·날짜·시간 등)

5코드 구조 원칙 — 비즈니스 규칙은 한 곳에

코드를 4개 층으로 나눈다. 비즈니스 규칙은 가장 안쪽 층에만 두고, 바깥 층은 DB·외부 API 같은 도구를 다룬다.

왜 이렇게 나누나

비즈니스 규칙이 DB 모델이나 API 응답 형태에 섞이면, 규칙 하나 바꾸는데 세 군데를 동기화해야 한다. 안쪽에만 두면 DB 바꾸든 LLM 바꾸든 비즈니스 규칙은 그대로 유지.

🚨 절대 규칙: 비즈니스 규칙(예: "메인 봇은 1명만") 절대 DB 모델이나 API 응답 형태에 박지 말 것. 가장 안쪽 층에만 둔다. 새 기능 만들 때 항상 ① 규칙 정의 → ② 추상 인터페이스 → ③ DB·외부 호출 구현 → ④ 흐름 조립 → ⑤ API 노출 순서.

내부 명칭 (개발자용): 가장 안쪽 = domain · 외부 시스템 연결 = infrastructure · 협업 흐름 = application · 외부 노출 = api. 각 층은 자기보다 안쪽에만 의존한다.

변수 namespaceSkill별 prefix 강제 (예: reservation.date, refund.reason) — 충돌 방지
Tool 권한 경계Tool은 Tenant(고객사) 단위로 묶임. Bot은 allowed_tool_ids 화이트리스트로 사용 가능 도구 제한

논의 필요 — PM 의견 받기

아래 6가지가 합의되면 설계 확정. 각 항목마다 추천안 표시.

🔴 분기 이탈 정책

상황: 환불 봇 진행 중 "아 그게 아니고 결제 변경"이라고 말함.

A. 무조건 메인 복귀 후 재라우팅
B. 다른 서브로 직접 점프 허용
C. 진행 중인 절차 끝까지 강제 (이탈 금지)

🔴 공통 규칙 우선순위

상황: 통화 단위 공통 규칙("상담사")과 봇 안 공통 규칙이 동시 매치.

A. 좁은 범위 우선 (봇 단위)
B. 넓은 범위 우선 (통화 단위)
C. 명시적 우선순위 필드

🔴 서브 봇 종료 시 동작

상황: 환불 봇이 절차 완료하고 end 도달.

A. 자동 메인 복귀 / B. 통화 자동 종료 / C. 명시적 신호 (현재 코드)
D. 멤버십에 정책 설정 (추천)

D 예시: 환불봇→return_to_main, 취소봇→end_call, 설문봇→transfer_to:확인봇. 기본값 자동 메인 복귀.

🔴 핸드오프 시 대화 history

상황: 메인이 사용자와 대화 중 환불봇으로 핸드오프.

A. 전체 history 그대로
B. 비우고 한 줄 컨텍스트만
C. 메인이 요약해 전달 (추천)

복귀 시엔 서브 대화를 메인 history에 합쳐 자연스러운 후속 대화.

🔴 의도 중첩 처리

상황: "예약 취소하고 환불도 되나요?" — 한 발화에 두 의도.

B. 명확화 질문 (MVP 추천)
C. 분해 + 순차 처리 (Phase 2)
D. 우선순위 + 후속 확인

B: 메인이 "어느 것부터?" 되묻기. 안전 100%. 운영하며 빈도 보고 C/D로 진화.

🔴 고객사(Tenant) 레벨 설정 범위

상황: 영업시간·금칙어·상담사 연결 정책이 봇마다 같음.

A. Tenant 레벨로 올림 (봇들이 자동 상속)
B. 봇 레벨 유지 (자유도↑ 중복↑)
C. 영업시간·금칙어=Tenant, voice·greeting=통화 단위 (분리, 추천)
