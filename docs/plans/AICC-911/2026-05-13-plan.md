## 1. 티켓 원문

### 배경/목적
현재 봇 설정(페르소나, 스킬, 툴, 플로우)은 DB 에 저장되나 변경 이력이 없다. 운영 중 잘못된 설정 배포 시 이전 상태로 복구할 방법이 없어 버전 관리 체계가 필요하다.

### 작업 내용
- 봇 설정 변경 시 스냅샷 저장 (`bot_config_snapshots` 테이블 또는 JSON 컬럼)
- API: `POST /bots/{id}/publish` — 현재 설정을 "배포됨" 버전으로 태깅
- API: `GET /bots/{id}/versions` — 버전 목록 조회
- API: `POST /bots/{id}/rollback/{version}` — 특정 버전으로 복원
- UI: 봇 설정 페이지에 "배포" 버튼 + 버전 히스토리 표시

### 완료 기준
- 설정 변경 후 rollback 시 이전 상태로 정확히 복원됨
- 버전 목록에 변경자, 변경 시각 표시
- 운영 중인 통화에 즉시 적용 vs. 다음 통화부터 적용 동작 명확히 정의

## 3. 설계 (신규)

### 3.1 도메인 모델 (Clean Architecture [[feedback_clean_architecture]])

```
domain/
  entities/bot_config_version.py   # 불변 스냅샷, 비즈니스 invariant
  repositories/bot_version_repo.py
infrastructure/
  models.py + BotConfigVersionORM
application/
  use_cases/publish_bot_config.py
  use_cases/rollback_bot_config.py
  use_cases/list_bot_versions.py
api/
  routers/bots.py + 3개 엔드포인트
```

### 3.2 엔티티

```
BotConfigVersion
  id: UUID
  bot_id: UUID
  version: int                 # 봇별 순차 증가 (1, 2, 3, ...)
  snapshot: JSONB              # 봇 설정 전체 한 덩어리
  label: str | None            # "프로덕션-2026-05-13" 등 사용자 입력
  published_at: datetime
  published_by: str            # 운영자 (1인 운영이라 항상 동일, 미래 대비)
  is_active: bool              # 현재 "운영 중" 버전 — bot 당 1개만 true
  notes: str | None
```

**스냅샷 범위** (한 JSONB 덩어리에 포함):
- Bot 자체: `voice_rules`, `llm_model`, Agent Type, prompt 등
- CallbotAgent: `global_rules`, `dtmf_map`, `pronunciation_dict` (또는 분리 후 `tts_pronunciation`/`stt_keywords`), `tts_speaking_rate`/`tts_pitch`, `idle_*`, `greeting_barge_in` (910 의 필드들)
- CallbotMembership 전체 트리: branch_trigger, silent_transfer (908)
- BotTagPolicy: `allowed_tag_ids` (912)
- 플로우(FlowEditor 값) 전체

### 3.3 결정 — 운영 중 통화에 즉시 적용 vs 다음 통화부터 (티켓 완료 기준 명시)

| 옵션 | 설명 |
|---|---|
| A | 진행 중 통화에는 그대로, 다음 통화부터 새 버전 |
| B | 진행 중 통화도 다음 turn 부터 새 버전 |

**권고: A (다음 통화부터)** — §5 #1
- 통화 중 페르소나/플로우 교체는 컨텍스트 단절 (AICC-908 의 sub 봇 인계 의도와 충돌)
- 1인 운영자가 통화 중 실수로 publish 했을 때 안전망
- 데이터: `CallSession` 에 `bot_config_version_id: UUID` FK 추가 → `voice_session` 시작 시 `bot.active_version_id` 를 한 번 스냅샷, 통화 끝까지 그 버전 사용
- AICC-908 §3.5 "변경 즉시 반영 vs 다음 통화" 와 같은 정책

### 3.4 결정 — 스냅샷 저장 방식 (티켓 본문: "테이블 또는 JSON 컬럼")

**권고: 별도 테이블 (`bot_config_versions`)** — §5 #2
- 한 봇이 수십 버전 누적 시 `bots` 테이블 row 크기 폭증 회피
- Postgres JSONB 1 컬럼으로 충분 — 인덱스 필요 시 `jsonb_path_ops` GIN

### 3.5 결정 — Rollback 동작
**권고: 신규 버전으로 복제** — §5 #3
- 버전 5 → 버전 3 으로 롤백 시 버전 6 = 버전 3 복제, `is_active=true` 이동
- 이유: 감사 로그 보존. 버전 3 자체를 다시 활성화하지 않음 (어떤 시점에 누가 롤백했는지 row 로 남음)

### 3.6 API

```
POST   /api/bots/{bot_id}/publish        { label?, notes? }
       → 현재 작업 중 설정(draft)을 새 BotConfigVersion 으로 freeze + is_active 이동
GET    /api/bots/{bot_id}/versions       → list (역순), version, label, published_at, published_by
GET    /api/bots/{bot_id}/versions/{v}   → 단일 스냅샷 (전체 JSONB)
POST   /api/bots/{bot_id}/rollback/{v}   → 해당 버전을 새 버전으로 복제 + is_active 이동
```

### 3.7 UI

- 봇 설정 페이지 (`/bots/[botId]/settings/page.tsx`):
  - 우상단 `Save (draft)` ↔ `Publish` 2개 버튼 분리
    - `Save`: 현재 동작 (draft 갱신 — bot 본체에 그대로 쓰기)
    - `Publish`: `Save` 후 `POST /publish` 호출
  - 사이드 패널 또는 모달: "버전 히스토리" — 각 row: 버전번호, label, published_at, published_by, [되돌리기] 버튼
  - 현재 활성 버전 배지
- 통화 상세 페이지:
  - 헤더에 "사용된 봇 버전 v.N" 표시 (AICC-912 의 태그 분석에도 디버깅에 도움)

## 4. 마이그레이션 (Alembic — AICC-907 baseline 이후)

신규 테이블: `bot_config_versions` (위 §3.2 스키마).
- UNIQUE: `(bot_id, version)`
- `is_active=true` 가 봇당 1개만이도록 partial unique index: `WHERE is_active = true`

기존 `bot_sessions` / `call_sessions` 에 FK 컬럼 추가: `bot_config_version_id` (nullable, 기존 통화는 NULL).

**초기 버전 백필**: 본 PR 배포 시 모든 봇에 대해 현재 설정으로 v1 row 생성, `is_active=true`.

## 6. 마일스톤 출구 기준
- Publish → 새 버전 row 생성, `is_active` 이동
- Rollback → 신규 row (이전 버전 복제), 이전 버전 row 보존
- 진행 중 통화는 publish/rollback 영향 받지 않음 (자동 테스트)
- 통화 종료 후 `CallSession.bot_config_version_id` 가 사용된 버전 정확히 가리킴
- 버전 목록에 published_at, published_by 표시
