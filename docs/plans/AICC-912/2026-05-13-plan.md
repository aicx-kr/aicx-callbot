## 1. 티켓 원문

### 배경/목적
현재 통화 종료 후 `post_call.py` 에서 세션 분석이 이루어지나 태그 시스템이 없다. 태그가 있어야 통화 유형별 분류, 빈도 분석, 지표 대시보드 연동이 가능하다.

### 작업 내용
- 태그 도메인 모델: `Tag(id, tenant_id, name, color)`
- 통화 종료 시 LLM 기반 자동 태깅 (예: "환불문의", "예약변경", "호전환")
- 수동 태그 추가/삭제 API (`POST /calls/{id}/tags`, `DELETE /calls/{id}/tags/{tag}`)
- 봇별 태그 정의 (어떤 태그를 자동 감지할지 설정)
- UI: 통화 상세 페이지에 태그 표시 + 태그 필터 검색

### 완료 기준
- 통화 종료 후 자동 태그 1개 이상 생성됨
- 태그로 통화 목록 필터링 가능
- 존재하지 않는 태그명 자동 생성 vs. 허용 목록 제한 정책 결정됨

## 3. 설계 (신규)

### 3.1 도메인 모델 (Clean Architecture [[feedback_clean_architecture]])

```
domain/
  entities/tag.py             # Tag, CallTag, BotTagPolicy - 비즈니스 invariant
  repositories/tag_repo.py    # 추상 인터페이스
infrastructure/
  models.py + TagORM, CallTagORM, BotTagPolicyORM
  repositories/tag_repo_impl.py
application/
  use_cases/auto_tag_call.py  # post_call analysis 결과 → CallTag 생성
  use_cases/manage_call_tags.py
  use_cases/manage_bot_tag_policy.py
api/
  routers/tags.py
```

### 3.2 엔티티

```
Tag
  id: UUID
  tenant_id: str        # AICC-909 의 "default" 기본값과 동일 정책
  name: str             # "환불문의", "예약변경", "호전환"
  color: str            # "#hex" or palette key
  is_active: bool

CallTag  (다대다 연결 + 출처)
  call_session_id: UUID
  tag_id: UUID
  source: enum { "auto", "manual" }
  created_at: datetime
  created_by: str | None    # manual 시 사용자 ID

BotTagPolicy  (봇별 자동 태그 정의)
  bot_id: UUID
  allowed_tag_ids: [UUID]   # 이 봇이 자동 감지할 태그 풀
```

### 3.3 자동 태깅 흐름
`post_call.py:42-83` `analyze_session()` 의 LLM 호출 응답 스키마에 `tags: [str]` 필드 추가 (분리 호출 X — 비용·지연 절감).
1. 응답의 태그명 리스트를 받음
2. `BotTagPolicy.allowed_tag_ids` 에 등록된 태그만 필터 (정책: 허용 목록 제한 — §5 #1)
3. 매칭된 태그를 `CallTag(source="auto")` 로 저장

### 3.4 결정 — 존재하지 않는 태그명 처리 (티켓 완료 기준 명시)
| 옵션 | 설명 |
|---|---|
| A | LLM 이 새 태그명 뱉으면 자동 생성 (`is_active=true` 새 Tag + CallTag 연결) |
| B | 허용 목록 제한 — `BotTagPolicy` 에 없는 태그는 버림 |

**권고: B**
- A 는 운영자가 의도하지 않은 태그 폭증 위험 — 1인 운영에서 관리 비용 증가 ([[project_solo_operation]])
- B 는 봇 콘솔에서 운영자가 명시적으로 태그 풀 정의 → tenant-specific 설정은 코드 X 콘솔 ([[feedback_no_hardcoded_tenant_config]])
- 향후 A 로 전환은 쉽지만 역방향은 어렵다 (이미 생성된 태그 정리 필요)

### 3.5 API

```
POST   /api/calls/{call_id}/tags                  { tag_id }   # 수동 추가
DELETE /api/calls/{call_id}/tags/{tag_id}                      # 수동 제거
GET    /api/calls?tag_id=<id>&tag_id=<id>                      # 필터 검색 (AND)
GET    /api/bots/{bot_id}/tag-policy                           # 봇 자동 태그 정의 조회
PUT    /api/bots/{bot_id}/tag-policy              { tag_ids }  # 봇 자동 태그 정의 수정
GET    /api/tags                                               # 테넌트 전체 태그 목록
POST   /api/tags                                  { name, color }  # 신규 태그
PATCH  /api/tags/{tag_id}                         { ... }      # 색·이름 변경
DELETE /api/tags/{tag_id}                                      # soft delete (is_active=false)
```

### 3.6 UI
- 통화 목록 (`frontend/src/app/bots/[botId]/calls/page.tsx`):
  - 칼럼 "태그" 추가 — Tag.color 칩 multi
  - 헤더에 태그 멀티 셀렉트 필터 (AND)
- 통화 상세 (`frontend/src/app/bots/[botId]/calls/[sid]/page.tsx`):
  - 상단 태그 칩 + "+" 버튼으로 수동 추가 모달
  - 모달: 기존 태그 선택 또는 (운영자 권한 시) 신규 태그 생성
- 봇 설정 페이지 (`/bots/[botId]/settings/page.tsx`):
  - "자동 태그" 섹션 신설 — `BotTagPolicy.allowed_tag_ids` 멀티 셀렉트 편집

## 4. 마이그레이션 (Alembic — AICC-907 baseline 이후)

신규 테이블 3개:
- `tags` (Tag)
- `call_tags` (CallTag — composite PK: call_session_id + tag_id)
- `bot_tag_policies` (BotTagPolicy — bot_id PK)

`call_sessions` 추가 인덱스 (필터 검색 성능): `(bot_id, ended_at desc)` 가 이미 있으면 그대로, 없으면 추가.

`Tag.tenant_id` UNIQUE 제약: `(tenant_id, name)` — 같은 테넌트 내 중복 이름 방지.

## 6. 마일스톤 출구 기준
- 통화 종료 후 정책에 등록된 태그 ≥ 1개 자동 생성 (해당 통화 트랜스크립트에 매칭되는 경우)
- 통화 목록에서 태그 필터 (AND) 동작
- 허용 목록 외 태그는 LLM 이 제안해도 무시됨 (로그에 "rejected" 기록)
- 수동 추가/삭제 동작 (UI + API)
