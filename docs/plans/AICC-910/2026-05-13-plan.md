## 1. 티켓 원문

### 배경/목적
`voice_session.py` docstring 에 barge-in 이 설계 의도로 명시되어 있고 Silero VAD 어댑터도 완성됐으나, speaking 상태에서 VAD speech_start 이벤트 수신 시 TTS 오디오 스트림을 실제로 중단하는 로직이 구현되어 있지 않다.

### 작업 내용 (티켓 원문 + 확장)
- (a) speaking 상태 중 VAD start 이벤트 → TTS 스트림 즉시 취소 + listening 상태 전환
- (a) 최소 barge-in 감도 튜닝: `vad_min_speech_ms` 파라미터 실험 (기본 200ms)
- (a) 오프닝 멘트(greeting) 중 barge-in 허용 여부 옵션 추가
- (b) 통화 무응답 자동종료 (확장)
- (c) DTMF (확장)
- (d) 키워드 보정 (확장)
- (e) TTS 발화 속도 조절 (확장)
- (f) STT/LLM/TTS 성능 개선 및 고도화 (확장)

### 완료 기준 (티켓 원문)
- TTS 재생 중 발화 시작 → 200ms 이내 TTS 중단
- Barge-in 후 STT 인식 정상 동작
- 확장 항목 출구 기준은 본 문서 §3

## 3. 서브 항목별 설계

### 3.1 (a) Barge-in — 잔여
- TTS 취소 직후 `state = "listening"` 명시 전환 (현재 cancel 후 다음 입력 대기에 위임)
- 봇 설정 `greeting_barge_in: bool` (기본 false). 콘솔 토글 — tenant-specific 은 코드 X ([[feedback_no_hardcoded_tenant_config]])
- 튜닝 매트릭스: `vad_min_speech_ms` ∈ {100, 150, 200, 300, 400} × {빠른 발화, 느린 발화, 잡음 환경}. 결과를 `AWKWARDNESS_LOG.md` 에 기록
- 자동 테스트 (`backend/tests/`): 모의 VAD start → 200ms 이내 `speech_task.cancel()` 호출 검증

### 3.2 (b) 통화 무응답 자동종료
**왜**: 사용자 침묵 시 회선·LLM 비용 누적, 통계 왜곡 방지 — 콜센터 표준.

**필드 (봇 설정, `CallbotAgent` 확장)**:
- `idle_prompt_ms: int` (기본 7000) — 이 시간 침묵 시 재안내 1회
- `idle_terminate_ms: int` (기본 15000) — 누적 침묵 시 자동 종료
- `idle_prompt_text: str` (콘솔 입력, 기본 "여보세요?")

**카운트 기준점**: 마지막 사용자 발화 종료 시각(`_on_speech_end`) 또는 봇 발화 종료 시각 중 늦은 것.

**end_reason enum 표준화** (`models.py:213` 자유문자열 → Literal):
```
end_reason ∈ {
  "normal",            # 사용자 정상 종료
  "idle_timeout",      # 무응답 자동 종료 (본 항목)
  "transfer_handoff",  # 상담사 호전환 후 클로즈
  "bot_terminate",     # 플로우 종료 노드
  "error",             # 예외 종결
  "client_disconnect"  # WebSocket 끊김
}
```
도메인 엔티티에 Literal/Enum ([[feedback_clean_architecture]]), ORM 은 str. 기존 자유문자열은 "normal" 로 backfill. enum 정의는 AICC-909 와 공유.

### 3.3 (c) DTMF
**입력 채널**: WebSocket 메시지에 `{"type": "dtmf", "digit": "1"}` 프레임 정의. 음성 클라이언트 SDK / SIP gateway 가 키패드 입력을 이 형태로 전달.

**핸들러**: `voice_session.py` 에 `_on_dtmf(digit: str)` 추가 — 현재 활성 `CallbotAgent.dtmf_map` 에서 키 룩업 → action 실행.

**action 종류**:
- `transfer_to_agent: <bot_id>` (AICC-908 의 인계 경로 재사용)
- `say: <text>` 후 listening 상태
- `terminate: <reason>` 종료 (`end_reason = reason`)
- `inject_intent: <intent_name>` LLM 컨텍스트에 시스템 메시지 주입

**UI 확장**: 현재 KVEditor 는 `"1" → "예약 변경"` 단순 텍스트. 별도 `DTMFActionEditor` 컴포넌트로 교체 — 우측에 action_type 드롭다운 + payload 입력. 데이터 형태:
```json
{ "1": { "type": "transfer_to_agent", "payload": "bot_xyz" } }
```

### 3.4 (d) 키워드 보정 — 발음사전 분리
**문제**: 현재 `pronunciation_dict` 한 dict 에 두 의도가 섞임 ("FTU" → "에프티유" 한글 발음 표기 vs "Awarefit" 같은 도메인 키워드 부스트).

**분리 모델**:
- `CallbotAgent.tts_pronunciation: dict[str, str]` — TTS 텍스트 치환용 (서브 e 의 SSML 도입 시 `<phoneme>`)
- `CallbotAgent.stt_keywords: list[str] | dict[str, float]` — STT phrase hint, optional boost

**STT 어댑터**: `google_stt.py` 의 `RecognitionConfig.speech_contexts=[SpeechContext(phrases=keywords, boost=10.0)]`. boost 기본 10.0.

**UI**: 봇 설정 페이지에 두 섹션 분리. 마이그레이션: 기존 `pronunciation_dict` 는 `tts_pronunciation` 으로 이전, `stt_keywords` 는 빈 배열 시작.

### 3.5 (e) TTS 발화 속도 조절
**필드 (봇 설정)**:
- `tts_speaking_rate: float` (기본 1.0, 범위 0.5 ~ 2.0)
- `tts_pitch: float` (기본 0.0, 범위 -20.0 ~ 20.0)

**적용**: `google_tts.py` 의 `AudioConfig(speaking_rate=..., pitch=...)` 추가. plain text + AudioConfig 만으로 가능. SSML 도입은 (d) 의 `<phoneme>` 필요 시점에 함께 검토 — 본 티켓 1차 범위에서는 plain.

**UI**: 봇 설정 페이지 "음성규칙" 옆에 슬라이더 2개.

### 3.6 (f) STT/LLM/TTS 고도화 — 1차 3개
| # | 항목 | 현재 | 개선안 |
|---|---|---|---|
| f1 | STT interim 활용 | True 로 받지만 final 만 사용 | interim 으로 **사용자 발화 중인지** 더 빠른 판정 → barge-in 응답 단축 |
| f2 | LLM 첫 토큰 지연 | `gemini-2.5-flash` 기본 | flash-lite 옵션 노출, thinking 비활성, 시스템 프롬프트 축소 검토 |
| f3 | TTS 청크 크기 | 코드 split | 첫 청크 작게(200ms), 이후 크게 — TTFF 단축 |

**2차로 미루는 항목** (AICC-911 봇 버전 관리와 합류):
- 봇별 모델 선택 UI (`Bot.llm_model` 필드는 이미 존재)
- 봇별 VAD 침묵 감지 시간 노출

## 6. 마일스톤 출구 기준
- (a) TTS 재생 중 발화 → 200ms 이내 중단 (자동 테스트) + 오프닝 옵션 토글 동작
- (b) 7초 침묵 → 재안내 1회 + 15초 누적 침묵 → 종료, `end_reason="idle_timeout"`
- (c) DTMF "1" 수신 → `dtmf_map["1"].type` action 실행 (4가지 중 하나)
- (d) `stt_keywords` 등록 단어가 인식률 개선 (정성 5단어 × 10회 비교 매트릭스)
- (e) TTS 출력의 speaking_rate / pitch 가 봇 설정대로 적용 (수동 확인)
- (f1) interim 활용 시 barge-in TTFF 단축 측정값 확인 (AICC-909 latency 분해 의존)
- (f2) flash-lite vs flash 응답 시간 비교 결과 기록 → `AUTO_LOOP_LOG.md`
- (f3) 첫 TTS chunk 송신 시점이 평균 단축 (TTFF 로그)
- 회귀 없음: 기존 단위 테스트 + barge-in 테스트 모두 통과
