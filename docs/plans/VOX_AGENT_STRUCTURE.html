Part 1 · 분석

vox 구조 역추정
및 설계

vox.ai 공식 docs의 노드 인덱스(11종)와 빌드 영역 구조만으로 vox 콜봇 에이전트의 내부 동작과 데이터 모델을 역추정한다.

다룰 내용: 전체 구조 · 두 모드 · 11 노드 · △△치과 예시 · 변수 시스템 · 멀티 에이전트

0. 전체 구조 한 장 — 콜봇 에이전트 아키텍처

vox에서 도출한 구조를 우리 aicx-callbot에 매핑한 레이어드 아키텍처. 이후 모든 섹션은 이 그림의 한 부분을 확대해 다룬다.

L1~L6의 6개 레이어. 굵은 박스는 4 자산, 점선 박스는 Bot의 평면 설정. 가운데 노란 레이어(L3)가 vox와 우리 모두에서 가장 차별화되는 부분.

1. 큰 그림 — 통화 한 통의 전체 흐름

전화 채널 → 음성 변환 → 에이전트 두뇌 → 통화 후 분석. 진짜 복잡함은 가운데 "두뇌"에 있다.

한 통화는 4개 레이어를 순차로 통과한다. 음성↔텍스트 변환은 컨테이너에 불과.

2. 두 가지 에이전트 모드

모드 A 프롬프트 에이전트

한 덩어리의 프롬프트로 봇 정의. LLM이 알아서 분기·도구 호출.

적합: 친절한 상담, FAQ, 일반 응대

장점: 빠르게 만듦, 표현력 자유

단점: "필드 누락" 같은 LLM 실수 종종 발생

모드 B 플로우 에이전트

노드를 선으로 잇는 그래프. 모든 분기·변수·전이를 명시.

적합: 신청서, 결제, 본인확인 (SLA 중요한 곳)

장점: 결정성·디버깅 쉬움 ("어느 노드에서 막혔는지" 보임)

단점: 그리는 시간이 든다

비교표

프롬프트플로우
정의 방법큰 프롬프트 1개노드 그래프
분기 처리LLM이 알아서condition 노드 (명시)
변수/슬롯LLM이 자유롭게extraction 노드 (강제)
턴당 LLM 호출1회여러 번 (extract+condition+conv)

통제력낮음높음

3. 11 노드 — 플로우 모드 빌딩블록

각 노드는 "받는 입력 → 하는 일 → 다음으로 넘기는 것"이 정해져 있다. 5개 카테고리로 묶어 외우면 쉽다.

A. 흐름 제어 (3종)

begin — 진입점, 첫 메시지 발화

end — 종료점, 종료 사유 기록

global — 어디서든 트리거되는 인터럽트 핸들러

B. 대화 (1종 · 가장 중요)

conversation — 여러 turn을 묶은 미니 대화 단위. 이 노드만 LLM을 직접 호출.

노드 안에 prompt + transitions(다음으로 가는 조건) 정의.

C. 데이터 처리 (3종)

extraction — 발화에서 슬롯 채우기 (예: "내일 3시"→date·time)

condition — 변수 비교로 분기 (age≥19 등)

api — REST 호출, 응답을 변수에 저장

D. 도구 호출 (1종)

tool — 등록한 API 도구 또는 내장 도구(end-call, sms, dtmf, transfer-call, transfer-agent)를 명시적으로 실행.

E. 외부 채널 (3종)

transfer-call — 사람 상담사 연결 (SIP refer)

transfer-agent — 다른 에이전트로 위임 ← 멀티 에이전트의 핵심

send-sms — 문자 발송

💡 두 가지 핵심 (카테고리 아님 · 인사이트)

global 노드는 어느 상태에서든 발동. "상담사 바꿔줘" 같은 인터럽트 표현용.

transfer-agent는 단일 거대 봇 대신 전문 봇 체인을 가능케 한다.

A~E 5개 카테고리의 노드 합 = 3+1+3+1+3 = 11종. DTMF는 노드가 아니라 conversation 설정 + send-dtmf builtin tool.

4. 구체 예시 — △△치과 예약봇 (1) 프롬프트 버전

시나리오: 인사 → 예약/문의 분기 → 예약이면 [날짜·시간·증상] 받기 → 등록 → 확인 SMS → 종료. "상담사" 발화는 어디서든 사람 연결.

프롬프트 에이전트 — 한 덩어리로 푼다

⚠️ LLM 한 번 호출에 모든 판단을 맡김. 빠르게 만들 수 있지만, "증상을 안 물어봤네" 같은 누락이 가끔 발생.

4. 구체 예시 — △△치과 예약봇 (2) 플로우 버전

✅ 분기·검증·재시도 모두 명시적. 누락 안 생기지만 그리는 시간이 더 든다. SLA 중요한 도메인(결제·본인확인)은 플로우 강제.

4-3. 한 턴(turn)이 어떻게 도는가

사용자가 "내일 3시요"라고 발화한 순간. 플로우 모드 기준 5단계 사이클.

이 다섯 단계가 매 턴마다 반복된다.

5. 변수 시스템 — 3종 출처

vox 에이전트의 핵심 도메인 모델 — 별도 자산처럼 다룬다. 프롬프트뿐 아니라 condition·API body·SMS 본문에서 모두 사용.

6. 멀티 에이전트 · 한 통화 안에서 모드 전환

transfer-agent는 노드이자 동시에 builtin tool. 두 모드 모두 다른 모드 에이전트를 호출 가능 → 통화 한 통 안에서 자연스러운 전환.

허브-앤-스포크 패턴 (실전 최다 사용)

한 통화 안에서 모드가 두 번 바뀌는 시나리오

턴실행모드
1사용자 "여보세요" → 메인봇 인사프롬프트
2"내일 예약하려고요" → LLM이 transfer_agent("예약봇") 호출프롬프트 → 플로우
3-5예약봇이 날짜·시간·증상 받기 → api 등록 → SMS플로우
6예약봇 end → transfer-agent로 메인봇 복귀플로우 → 프롬프트
7"가격이 얼마예요?" → RAG로 답변프롬프트

사용자는 모드 차이를 전혀 느끼지 못한다. 백엔드에서 두 번 핸드오프가 일어났을 뿐.

6-3. 가장 흔한 실전 패턴 — 프롬프트 메인 + 분기별 플로우

vox 사용자 80%가 쓰는 패턴. 메인은 프롬프트로 자유 응대, 구조화 필요한 분기만 플로우. UI는 통합, 데이터는 분리.

어드민 UX — 빌더가 보는 한 화면

내부 데이터 구조 — Skill 분리 + global 룰

💡 핵심: UI는 통합("메인 + 분기들" 한 화면), 데이터는 분리(Skill 4개). global 룰이 라우팅 담당 → prompt는 자유 응대만, 라우팅 변경 시 prompt 안 건드림.
Part 2 · 비판

vox 구조의
문제점

vox는 잘 만들어진 매니지드 플랫폼이지만, 우리가 그대로 따라가면 안 되는 5가지 구조적 한계가 있다. 이 한계가 Part 3 개선안의 출발점.

다룰 내용: prompt 라우팅 박힘 · Skill 부재 · 변수 자유형 · 모드 인지부담 · 운영설정 분산

7. vox가 갖는 5가지 한계

각 한계 옆에 우리가 어떻게 풀지 한 줄로 요약. 자세한 설계는 Part 3에서.

① prompt에 라우팅 로직이 박힘

문제: "예약이면 transfer_agent 호출하라" 같은 라우팅 지시를 system prompt에 적어야 함. prompt가 길어지고, 라우팅 변경 시 prompt 수정 필요. 자유 응대 톤과 라우팅 로직이 한 덩어리에 섞임.

→ 우리의 해법: global 룰을 별도 영역으로 분리. prompt는 톤·페르소나만 담당, 라우팅은 globals[]가 담당.

② 재사용 단위 모호 — Skill 부재

문제: vox는 agent가 최소 단위. 같은 결제 로직을 5개 봇에서 쓰려면 5번 복제해야 함. 복제본 간 drift 발생 → 한 봇 고치면 나머지 4개 누락.

→ 우리의 해법: Skill을 1급 도메인으로 승격. Bot ↔ Skill N:N 관계. 워크스페이스 단위 자산 카탈로그.

③ 변수 스키마 자유형

문제: extraction이 dict[str, Any]에 던지는 식. 타입 검증·자동완성 없음. "date"인지 "appointment_date"인지 노드별로 표기 다르면 무용지물. 디버깅 시 어디서 채워졌는지 추적 어려움.

→ 우리의 해법: Skill에 variable_schema 선언 (Phase 2). dynamic·system·extracted 출처별 namespace 분리.

④ 두 모드의 인지 부담

문제: 사용자가 "이 봇은 프롬프트 모드인가 플로우 모드인가" 의식해야 함. 한 봇 안에서 두 모드 혼용 불가 — 무조건 transfer-agent로 별도 봇 만들어야. 작은 분기에도 신규 agent 생성 부담.

→ 우리의 해법: 단일 graph executor. 프롬프트 = 1-노드 그래프. 두 모드는 UI 토글만, 데이터는 동일 스키마.

⑤ 운영 설정이 여기저기 흩어짐

문제: 음성 관련만 봐도 build/voice/voice-select, build/voice/pronunciation-guide, build/conversation/speech-settings, build/conversation/first-message, build/conversation/dtmf 5군데 분산. 빌더가 어디서 무엇을 설정하는지 헷갈림. 운영자가 동일 봇의 발음 사전을 찾을 때 3분 걸림.

→ 우리의 해법: Bot 평면 설정으로 통합 — 한 화면에 voice·greeting·발음사전·DTMF·인터럽트 정책 모두 노출. 자산(Persona/Skill/Knowledge/Tool)과 명확히 분리.

핵심 통찰

vox의 한계는 "잘 만든 도구"의 한계가 아니라 "범용 매니지드 SaaS"의 한계다. 그들은 1만 고객을 위해 표현력·안전성·유지보수 사이 균형을 잡는다. 우리는 자체 도메인(우리 고객사 콜봇)에 맞춰 그 균형점을 다시 잡을 수 있다.

8. 우리 하이브리드 구조 — 한 눈에 보는 장단점

Prompt(메인) + Flow(분기) = 통합 에이전트. §6-3 패턴이 가져오는 7장점과 7단점, 그리고 단점 보완 매핑.

✅ 장점 7가지

① 인지 부담 0 — 사용자는 "그냥 봇". 빌더도 모드 의식 안 함

② 비용·지연 최적화 — 일반 응대 LLM 1콜, 분기 진입 시만 다중 노드

③ 점진적 강화 — 프롬프트로 시작 → 빠지는 케이스 발견 → 그 분기만 플로우

④ 재사용 — 결제 플로우를 N개 봇이 공유 (drift 방지)

⑤ 권한 분리 — 메인=CS팀, 결제=결제팀, Skill 단위 격리

⑥ 디버깅 격리 — "라우팅 실패" vs "분기 내부 버그" 책임 경계 명확

⑦ 다국어·페르소나 — 외국어 분기에 다른 페르소나 갈아끼움

⚠️ 단점 7가지

① 라우터 단일 실패점 — 메인 LLM 의도 오인 → 잘못된 분기

② 비결정성 — 같은 발화에 라우팅 결과 달라질 수 있음

③ 변수 인계 복잡 — 메인→분기→복귀 시 컨텍스트 누락 위험

④ 컨텍스트 분단 — 분기 안에서 "아 그게 아니고…" 발화 처리

⑤ 글로벌 충돌 — 메인·분기의 globals 우선순위 모호

⑥ 분석 통합 — 통화 1건이 여러 Skill 거침 → 집계 단위 모호

⑦ 버전 매트릭스 — 메인 v3 + 결제 v2 + 환불 v1 운영 조합 추적

언제 빛나는가 vs 부적합한 케이스

✅ 빛나는 케이스 (우리 B2B 콜봇)

• 의도 5~15개 (적당히 분기됨)

• 일부 분기만 strict SLA (결제·본인확인)

• 운영하며 점진적 강화 가능한 환경

• 고객사별 페르소나·다국어 갈아끼우기

❌ 부적합 케이스

• 모든 분기가 strict SLA → Bot.agent_type=flow 단일 그래프

• 의도 2~3개뿐 → 프롬프트 1개로 충분

• 일회성 단순 봇 → 오버엔지니어링

9. vox 분석 결론 · aicx-callbot 매핑

vox docs를 역설계해서 얻은 "vox는 왜 이렇게 만들었나" 5가지 통찰. 우리 적용 원칙은 §11-3에서 별도 정리.

vox 설계가 말해주는 5가지 통찰

① 프롬프트 = 1-노드 플로우

두 모드는 본질적으로 같은 그래프. vox는 UX·과금 분리 위해 둘로 보여주지만 엔진은 통합 가능하다는 뜻.

② 변수가 첫째, 프롬프트가 둘째

extraction·condition·api·sms 모든 노드가 변수를 통해 대화. 변수 모델이 빈약하면 vox-급 표현력 불가.

③ global은 상태기계 밖의 룰

"어디서든 상담사" 같은 인터럽트는 노드/엣지로 표현 불가. vox가 별도 개념(global node)을 둔 이유.

④ transfer-agent는 노드+tool 양방향

두 모드가 한 통화 안에서 매끄럽게 섞이는 비밀. 노드/tool 어느 쪽에서든 호출 가능해야 멀티 에이전트가 자연스러움.

⑤ 자산(재사용)과 운영 설정(평면)을 칼같이 분리

Persona·Knowledge·Tool은 라이브러리화. Voice·greeting·인터럽트 정책 등은 Bot에 박힌 평면 설정. 이 구분이 흐려지면 라이브러리 폭발 또는 봇별 중복 폭발.

10. graph-ready Skill 데이터 모델

핵심 아이디어: 모든 Skill은 그래프다. 프롬프트 모드는 노드 1개짜리 그래프일 뿐.

같은 스키마에 두 모드가 담긴다

프롬프트 모드 = 1-노드 그래프

플로우 모드 = N-노드 그래프

이 설계가 주는 이득

  • 마이그레이션 0 — 프롬프트 → 플로우 = 노드 추가만
  • 런타임 코드 1개 — graph executor 하나, 1-노드는 fast path
  • 버전 관리 단순 — Skill 단위 스냅샷, diff 깔끔
  • Frontdoor 핸드오프 통일 — transfer-agent는 결국 "Skill ID로 점프 + 변수 인계"

깔끔한 구조의 6가지 원칙

① 런타임은 하나, UI는 둘

graph executor 한 곳에 노드 타입별 핸들러 집중.

② Skill = Graph + refs

모든 Skill이 Graph 필드 보유. 프롬프트 모드는 1-노드 그래프.

③ VariableContext는 세션당 1개

3종 출처 합치고, 모든 노드·프롬프트·식에서 동일 참조.

④ Dispatcher가 매 턴 첫 단계

global 룰 평가가 executor 진입보다 먼저.

⑤ Frontdoor가 핸드오프 단일 진입점

transfer-agent는 노드든 tool이든 결국 같은 호출.

⑥ 같은 Skill을 두 시각으로

모드 토글이 데이터 변환 없이 뷰만 전환.

발표 메시지: 우리 aicx-callbot은 vox와 같은 표현력을 단일 graph executor로 달성한다. 두 모드를 별도 코드로 분리하지 않는 게 핵심 — UI만 두 뷰, 런타임은 하나.

12. 구조 개선안 비교 — 논의 포인트

설계 방향이 갈릴 수 있는 5가지 결정 지점. 각 옵션의 trade-off와 추천을 정리했다. 발표 후 논의에서 합의할 항목.

① 핸드오프 모델 — Skill 단위 vs Sub-graph 중첩

옵션 A · Skill 단위 핸드오프 추천

+ 모델 단순. 권한·버전 Skill 단위. vox와 1:1 호환.

+ Frontdoor 1개로 통일.

− 핸드오프 지점 약간 latency.

옵션 B · Sub-graph 중첩

+ Latency 0. 매끄러운 전환.

− 재귀 그래프 — 모델·UX 복잡.

− vox 마이그레이션 변환 필요.

② 조건 평가 엔진 — CEL vs JSONLogic vs LLM

옵션표현력안전성학습추천
CEL (Common Expression Language)높음 (Google 표준)샌드박스 ✓중간추천 산업 표준 · k8s admission 등 검증됨
JSONLogic중간샌드박스 ✓쉬움JSON UI에 친화적이지만 표현 한계
Python eval최고위험쉬움금지 — 인젝션 표면
LLM 자체 판정최고비결정적없음condition 노드용으로 부적합 · 비용·지연

③ FlowView 캔버스 라이브러리

옵션커스터마이징리스크추천
react-flow (Xyflow)높음 · 핸드러·미니맵 풍부중간 · 라이선스 MIT추천 표준 도구
자체 구현 (SVG/Canvas)최고높음 · 6주+ 비용v3 이후 검토
Mermaid (read-only)낮음 · 편집 불가낮음플로우 미리보기/문서용만

④ global 룰 매칭 방식

A · regex/keyword

+ 빠름 · 결정적

− "상담사 좀 부탁드려요" 같은 변형 놓침

B · LLM 의도 분류 추천

+ 자연어 변형 잡음

− 매 턴 LLM 1콜 추가 비용

→ 작은 분류 모델(haiku)로 비용 완화

C · 하이브리드

+ A로 1차 필터 → 의심되면 B

− 구현 복잡

→ Phase 3에 검토

⑤ Bot ↔ Skill 관계 — 1:N vs N:N

옵션 A · 1:N (Bot이 Skill 소유)

Skill은 봇 안에서만 의미 있음. 다른 봇이 쓰려면 복제.

+ 단순. 권한·버전 명확.

− "결제봇" 같은 공통 Skill을 여러 봇에서 못 씀. 복제 시 drift.

옵션 B · N:N (Skill 라이브러리) 추천

Skill은 워크스페이스의 자산. 여러 봇이 참조.

+ 재사용 · 라이브러리 가치.

− "삭제하면 다른 봇 깨짐" 같은 의존성 관리 필요.

→ 워크스페이스 단위 자산 카탈로그가 vox UX 핵심.

13. 추가 개선안 — 운영·확장

위 5개가 핵심 결정이라면, 아래는 "있으면 좋은" 개선안들. Phase별로 도입 시점만 합의하면 됨.

⑥ Skill 버전관리 — 스냅샷 vs Git-like

스냅샷: "v1, v2, v3" 단순 라벨. 구현 쉬움. 추천 (MVP).

Git-like: branch · merge · 3-way diff. 강력하지만 과합. v4+ 검토.

⑦ 테스트 콘솔 — 텍스트 우선 vs 음성 우선

텍스트: WebSocket 채팅. 빠른 반복. MVP

음성: 실제 STT/TTS 통과. Phase 2.

→ 둘 다 같은 SkillEditor 안에서 토글로 지원.

⑧ Variable 스키마 — 자유 vs 명시적

자유: dict[str, Any]. 빠른 시작.

명시적: Skill에 variable_schema 선언. 타입 체크·자동완성. 추천 (Phase 2)

⑨ Tool 호출 권한 — 공개 vs 봇별

공개: 모든 Skill이 모든 Tool 호출 가능. 단순.

봇별: Bot.allowed_tool_ids로 제한. 추천 사고 방지.

⑩ Post-call 추출 — 동기 vs 잡 큐

동기: 통화 종료 직후 인라인. 단순. MVP.

잡 큐: Celery/Arq. 대량·재시도·메트릭. Phase 2~3.

⑪ 관측성 — turn-level tracing

매 턴마다 latency·cost·노드 ID·LLM 토큰을 trace로. OpenTelemetry 권장.

→ Waterfall.tsx에 그대로 띄우면 디버깅 천국.

논의 우선순위 (제안)

우선결정 지점왜 지금 정해야 하나
P0① 핸드오프 (A vs B)도메인 모델 형태가 갈림
P0⑤ Bot↔Skill 관계 (1:N vs N:N)스키마·권한 설계 영향
P1② 조건 평가 엔진condition 노드 출시 전
P1④ global 매칭 방식Phase 3 시작 전
P2③ 캔버스 라이브러리FlowView 작업 시작 전
P2⑥~⑪ 운영·확장 개선안Phase별 도입 시점만
발표 후 액션: P0 두 개(① 핸드오프 / ⑤ Bot-Skill 관계)는 다음 미팅 전에 결정. P1·P2는 해당 Phase 킥오프 때 결정.

14. 정리 · Q&A

발표 한 줄 정리

vox 콜봇 에이전트는 "두 모드 × 4 자산 × 11 노드"로 완전히 분해된다. 두 모드는 사실 UI 구분일 뿐이고, 우리 aicx-callbot은 단일 graph executor + 공유 VariableContext + global dispatcher로 같은 표현력을 달성할 수 있다.
